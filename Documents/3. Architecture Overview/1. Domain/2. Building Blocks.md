## Business Rule
Business constraints to ensure domain integrity through workflows. 

Rules
- Inherit IBusinessRule Interface
## Dependency 
Interfaces to define contract for required dependencies that should be supplied to domain. The result of this dependencies should always validated by domain before usage.

Rules
- Inherit IDependency Interface
## Domain Service
A independent class to capture and model the pure behavior of a business workflow. In DDD this is encouraged to be done inside entities when possible.

Rules:
- Inherit IDomainService Interface
- Have One Public Method
## Entity
A class to first represent data model of domain. 
Depending on the model type it could also
- Model Business Behavior
- Verify Invariants (Business Rules)
- Validate input
## Value Object
More specific struct to capture and validate domain values. It ensures the existing rules on the values going through domain. It also makes sure operations on values result in another valid value. Therefore always comes with predefined rules to be checked on creation, and modification of this object.
## Command
A body of data to initiate a business workflow.
Should be handled by this domain.
## Event
Fire and forget side effects of processing operations in domain to notify others about the immutable permanent changes.
Only [[System Events and Business Events#Business Event|Business Event]] Belong Here.
## Specification
A condition to be checked on other blocks. It could be used as query filters.
