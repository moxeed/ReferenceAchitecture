**Message**, A body of data sending asynchronously from one party to another. Whether on network of inter-process.

**Command**, A message to initiated another action in other service. the sender is responsible for the fulfillment of transaction.

**Event**, A message to notify other of the immutable changes happened.

**Query**, A message to fetch data from other services.

**Distributable,** built as loosely coupled services, each performing a single function to support horizontal scalability.

**Observable,** allowing requests across multiple services to be tracked using built-in **monitoring**, **tracing**, and **logging** features.

**Portable,** not tied to specific vendors or implementations.

**Interoperable,** exposing functionality through **APIs**.

**Available,** gracefully handling service failures.

**Domain Driven Design**, to capture business domain as is through entities, aggregates, value objects, events, and domain services.

**Orchestration**, to handle the flow of operations and its integrity through a single unit. e. g. application services.

**Choreography**, handle flow and integration through events and commands in a distributed manner.

**Behavior**, refers to cross-cutting concerns that are applied automatically to Application Services/Use Cases through a pipeline pattern. Behaviors handle aspects like validation, logging, authorization, transactions, etc., without cluttering your core business logic.
